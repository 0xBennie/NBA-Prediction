# 机器学习与统计方法论

## 系统架构总览

```
Polymarket价格 ──→ ┌─────────────┐     ┌──────────┐     ┌──────────┐
                   │ 规则评分引擎 │ ──→ │ ML修正层 │ ──→ │ 推送决策 │
Pinnacle赔率  ──→ │  (0-100分)   │     │ (±15分)  │     │  + 风控  │
ESPN伤病/B2B ──→ └─────────────┘     └──────────┘     └──────────┘
CLOB真实价格 ──→                          ↑                   │
                                    训练数据回喂              ↓
                                         ↑              ┌──────────┐
                                    ┌────┴─────┐        │ Telegram │
                                    │ 结果回填  │←───── │  + 回测  │
                                    └──────────┘        └──────────┘
```

系统分两层：
1. **规则引擎**：基于体育博彩研究的确定性评分（0-100分）
2. **ML修正层**：在规则分基础上做±15分的动态调整

这种"规则+ML"混合架构的优势：冷启动时规则引擎独立工作，ML层随数据积累逐步介入优化。

## 第一层：规则评分引擎

### 1. Edge计算 — 跨市场价差

**原理**：Polymarket和传统博彩公司（Pinnacle等）对同一场比赛定价不同。当Pinnacle认为某队胜率70%，但Polymarket卖65¢，就存在5%的价值错配（edge）。

**公平概率提取 — Shin法去vig**：

博彩公司的赔率包含overround（利润边际），需要去除才能得到"公平概率"。我们使用Shin法而非简单归一化：

```
传统归一化: P_fair = (1/赔率) / Σ(1/赔率_i)
Shin法:     P_fair = [√(z² + 4(1-z)r²) - z] / [2(1-z)]
            其中 z = Σ(1/赔率_i) - 1 (overround)
                 r = 1/赔率 (隐含概率)
```

**为什么用Shin法**：传统归一化假设margin均匀分布在所有结果上，但实际上博彩公司的margin更多加在冷门上（因为内幕交易者更倾向投注冷门获取高赔率）。Shin模型考虑了这种信息不对称，对热门的去vig更保守，对冷门更激进。

**Edge计算**：
```
raw_edge = 公平概率 - Polymarket买入价
effective_edge = 公平概率 - (买入价 + 交易成本)
```

使用CLOB的`/price`端点获取真实买入价（ask），而非Gamma API的中间价，因为真实可成交价格通常比中间价低1-3%。

### 2. Edge评分（0-40分）— 数据校准

基于197场历史信号的回测发现：

| Edge区间 | 样本 | 胜率 | 结论 |
|---------|------|------|------|
| 0.5-1.0% | 75 | **64%** | 甜区，最佳信号 |
| 1.0-2.0% | 47 | 49% | 尚可 |
| 2.0-3.0% | 6 | 33% | 反向指标 |
| >3.0% | 5 | 33% | 噪声/定价异常 |

大edge（>3%）通常不是真正的价值机会，而是：
- 流动性差的市场（价格不反映真实供需）
- 数据延迟（赔率已经变了但缓存没更新）
- 定义不一致（Polymarket和博彩公司的market不完全匹配）

因此评分函数呈**倒U型**：

```
edge 0.8-1.5%  → 30-40分（甜区，给最高分）
edge 1.5-3.0%  → 25-40分（递减）
edge >3.0%     → 15分封顶（可能是噪声）
edge 0.2-0.5%  → 0-15分（edge太小）
edge <0.2%     → 0分
```

### 3. 盈亏比评分（0-15分）— 风险管理

买入价格直接决定了盈亏比（risk/reward ratio）：

```
买入价P → 赢: ROI = (1/P - 1)    输: ROI = -100%
例: 买40¢ → 赢+150% / 输-100% → 盈亏比 1.5:1
    买70¢ → 赢+43%  / 输-100% → 盈亏比 0.43:1
    买80¢ → 赢+25%  / 输-100% → 盈亏比 0.25:1（一亏吃4赢）
```

评分逻辑：
```
40-55¢ → 15分（甜区：盈亏比0.8-1.5，胜率合理）
55-65¢ → 10分
65-75¢ → 5分
>75¢   → 0分 + 禁止推送（盈亏比太差）
<25¢   → 0分 + 禁止推送（冷门胜率太低）
```

### 4. 盘口移动（0-25分）— 跨市场响应差

**核心概念：Discordance（跨市场响应差）**

```
discordance = Pinnacle概率变化 - Polymarket价格变化
```

当Pinnacle的赔率向某个方向移动（比如因为伤病消息），但Polymarket还没跟上时，discordance为正值。这是我们的核心alpha来源——信息传播滞后。

评分：
```
|discordance| >= 5%  → 20分
|discordance| >= 3%  → 15分
|discordance| >= 2%  → 10分
|discordance| >= 1%  → 5分
快速变动(>5%/小时)   → 额外+5分
```

### 5. 伤病影响（0-20分）— RAPTOR Elo模型

使用ESPN伤病报告，通过RAPTOR值（球员综合贡献指标）量化伤病影响：

```
Elo惩罚 = Σ(球员RAPTOR值 × 状态乘数 × 12)
状态乘数: Out=1.0, Doubtful=0.75, Questionable=0.5, Probable=0.15

injury_delta = 主队Elo惩罚 - 客队Elo惩罚
正值 = 主队伤更重 → 买客队利好
负值 = 客队伤更重 → 买主队利好
```

评分根据买入方向调整后的effective_injury：
```
>50 → 20分（巨大伤病优势）
>30 → 15分
>15 → 10分
>5  → 5分
```

### 6. 背靠背 B2B（0-5分）

基于 Small 2007 (NESSIS) 研究：
- B2B导致约2.26分的分差影响
- 胜算乘数约0.62（vs 3+天休息）
- 用logistic模型换算：2.26分 ≈ 3.3%胜率变动

```
B2B胜率修正 = 2.26 / 17 × 0.25 ≈ 0.033 (3.3%)
```

仅当修正幅度>2%且方向对买入方有利时才加5分。

## 第二层：ML修正层

### 模型进化路径

```
数据量 < 30条  → ML不介入，输出0修正
30-100条      → Logistic Regression（强正则化 + 概率校准）
100+条        → LightGBM（限深度 + 概率校准）
```

### 特征工程（8维）

| 特征 | 含义 | 设计意图 |
|------|------|---------|
| `edge` | 有效edge（扣除成本后） | 核心定价信号 |
| `poly_price` | 买入价 | 捕捉价格区间效应 |
| `injury_delta` | 调整后的伤病差 | 伤病信息是否被市场充分定价 |
| `home_b2b` | 主队是否B2B | 疲劳因素 |
| `away_b2b` | 客队是否B2B | 疲劳因素 |
| `line_moved` | 盘口变化幅度（连续值） | 信息流强度 |
| `volume_ratio` | 24h成交量/基准值 | 市场关注度 |
| `source_quality` | 赔率源质量分 | 数据可信度 |

**设计原则**：特征应与市场预测"去相关"。ML不应重新学习"强队会赢"（这已经在赔率里了），而应学习"规则引擎还没捕捉到的残差模式"。

### Logistic Regression（30-100条数据）

```python
LogisticRegression(C=0.1, max_iter=1000)  # 强正则化
+ CalibratedClassifierCV(cv=5)            # 概率校准
```

- **C=0.1**（强正则化）：数据量小时防止过拟合，让模型偏保守
- **概率校准**：确保模型输出的概率是"真实概率"（如预测60%胜率，实际应该赢60%）。这对Kelly仓位计算至关重要——未校准的概率会导致严重的仓位错误

### LightGBM（100+条数据）

```python
LGBMClassifier(
    n_estimators=50,     # 少量树，防过拟合
    max_depth=4,         # 浅树
    min_child_samples=10,# 每个叶子至少10个样本
    learning_rate=0.05,  # 慢学习
)
+ CalibratedClassifierCV(method='isotonic')  # Isotonic校准
```

- **限深度+少量树**：体育预测数据量永远不会很大（每天5-10个信号），过复杂的模型一定会过拟合
- **Isotonic校准**：比Sigmoid校准更灵活，适合非线性概率映射

### 修正值映射

```python
win_prob = model.predict_proba(X)[0][1]  # 模型预测的胜率
adjustment = (win_prob - 0.5) × 30       # 映射到±15范围
adjustment = clamp(adjustment, -15, +15)
```

- 模型认为50%胜率 → 0修正（规则分已经足够）
- 模型认为65%胜率 → +4.5分修正
- 模型认为80%胜率 → +9分修正
- 模型认为35%胜率 → -4.5分修正（降低规则分）

### 训练数据来源

改进后使用`signal_log`全量数据（不只是推送的信号）：
- **标签**：`prediction_correct`（1=我们推荐的方向赢了，0=输了）
- **样本**：所有已结算信号，包括低分未推送的
- **好处**：ML能从"低分但赢了"和"高分但输了"的信号中学习，找到规则引擎的盲点

### 自动重训练

每积累10条新的已结算数据，自动触发重训练。模型保存为pickle文件（`ml_model_game.pkl`），服务重启时自动加载。

## 统计方法

### 1. Kelly仓位管理 — 不对称公式

传统Kelly公式假设对称赔率，但Polymarket是二元结算（赢=1/price，输=0）：

```
标准Kelly: f = p/b_loss - (1-p)/b_win
其中:
  p = 公平概率（我们估计的真实胜率）
  b_win = (1 - buy_price) / buy_price  （赢了赚多少倍）
  b_loss = 1.0  （输了亏100%）

实际使用: ¼ Kelly, 上限5%
```

**为什么用¼ Kelly**（Thorp 2007推荐）：
- 全Kelly波动率极大（50%回撤是常态）
- ¼ Kelly保留76%的长期增长率，但回撤降至原来的¼
- 上限5%防止单笔过大

### 2. 概率校准（Calibration）

**核心理念**：对预测系统来说，校准比准确率更重要。

```
准确率: 预测"赢"的有多少真的赢了
校准度: 预测"60%胜率"的信号，实际是否赢了60%
```

Kelly公式直接使用概率值计算仓位，如果概率不校准：
- 概率偏高 → 仓位过大 → 单笔亏损巨大
- 概率偏低 → 仓位过小 → 错过收益

使用`CalibratedClassifierCV`进行事后校准：
- 30-100条数据：Sigmoid校准（Platt Scaling）
- 100+条数据：Isotonic校准（非参数方法，更灵活）

### 3. 回测验证体系

**Signal Log全链路追踪**：
```
信号产生 → 记录特征/评分/买入方向
    → 比赛结束 → 回填实际结果
    → 计算两个维度:
        1. prediction_correct: 方向对不对？
        2. actual_roi: 实际盈亏（含费用）
```

**每日报表统计**：
- 近7天胜率、平均ROI
- 按评分/edge/赔率来源的分组表现
- 推送信号逐条结果列表

**统计分组分析**：
```sql
-- 按评分区间的胜率
SELECT score_bucket, COUNT(*), AVG(prediction_correct), AVG(actual_roi)
FROM signal_log GROUP BY score_bucket

-- 按Edge区间的胜率
SELECT edge_bucket, COUNT(*), AVG(prediction_correct), AVG(actual_roi)
FROM signal_log GROUP BY edge_bucket
```

这些分组数据用于持续校准评分阈值和权重。

### 4. 4级瀑布赔率源

```
优先级1: Pinnacle       (质量分1.0, 全球最sharp的博彩公司)
优先级2: DraftKings     (质量分0.85)
优先级3: FanDuel        (质量分0.85)
优先级4: 共识均值        (质量分0.7, 所有可用博彩公司的去vig均值)
```

非Pinnacle来源在评分中扣最多15分（`source_penalty`），确保低质量数据源不会产生高分信号。

## 风控体系

### 推送门槛（5重过滤）

```python
should_push = (
    final_score >= 50           # 综合评分达标
    and raw_edge > 0.4%         # 最低edge
    and effective_edge > 0      # 扣费后仍有利可图
    and buy_price >= 25¢        # 不推极端冷门（胜率太低）
    and buy_price <= 72¢        # 不推高价信号（盈亏比太差）
)
```

### 每日推送上限

最多10条/天，防止信息过载。24小时去重（同一比赛不重复推送）。

### 数据收集节奏

| 数据 | 频率 | 来源 |
|------|------|------|
| 比赛盘扫描 | 每小时 | Polymarket Gamma API + CLOB /price |
| 赔率获取 | 每小时（5分钟缓存） | The Odds API（Pinnacle + US博彩） |
| 期货盘扫描 | 每天08:00 | Polymarket Gamma API |
| 结果回填 | 每小时 | The Odds API /scores（免费端点） |
| 每日报表 | 每天09:00 | signal_log聚合 → Telegram |
| ML重训练 | 每10条新数据 | signal_log全量 |

## 已知局限与改进方向

1. **样本量**：当前仅26条去重后的已结算信号，ML层还未真正发挥作用（需30条触发首次训练）
2. **赔率覆盖**：The Odds API只覆盖赛前1-2天开盘的比赛，Polymarket上一周后的比赛无法评分
3. **伤病时效**：ESPN伤病报告有延迟，突发轮休可能来不及反映
4. **单向交易**：目前只做买入推荐，不做卖出/对冲建议
5. **没有赛中策略**：比赛开始后的价格波动不做追踪
